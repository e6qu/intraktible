# intraktible — Known Issues & Bugs

Tracked alongside `PLAN.md`; updated in the same PR at the end of every phase.
Format: `ID | severity | component | description | status`.

## Open (deferred / limitations during Phase 5)
- `D21b | low | store | resuming a durable store incrementally from Head (instead of a full rebuild from offset 0 on boot) is not implemented — it needs per-event idempotency or transactional apply+marker (the analytics stats counter is not idempotent), a consistency project. Full rebuild on boot is correct and fine at MVP scale (and the streaming WAL/SQLiteLog keep it bounded) | deferred (consistency)`

## Closed by decision (won't implement)
- `D9 | decision-engine | CEL conditions — won't implement. expr-lang already serves Rule/Split conditions + Assignment, and Starlark serves the Code node, so conditions/expressions are fully covered. A second expression engine (cel-go) would add a dependency and parallel semantics for no new capability. Revisit only if a concrete need for CEL specifically arises.`
- `D11 | decision-engine | batching the per-node decision events — won't implement. The per-node DecisionStarted → NodeEvaluated… → Completed/Failed stream IS the replayable, node-by-node decision history (PLAN §3.3); batching would trade that granularity for throughput the MVP does not need. A future high-volume optimization, not a bug.`

## Fixed
- `D21 | store | added a Postgres store.Store adapter (store.NewPostgres via jackc/pgx/v5 + pgxpool) behind the existing interface, selectable with serve --store=postgres (DSN from INTRAKTIBLE_POSTGRES_DSN) — the durable/shared option for large or multi-process projections. One docs(collection, key, doc JSONB) table, parameterized queries. VERIFIED against a real PostgreSQL 17 (not just written): the full Store-contract test (store_test.go TestPostgresStore, gated on INTRAKTIBLE_TEST_POSTGRES so the no-Postgres CI skips it) passes, and a real-binary --store=postgres smoke created a flow + read it back after a restart (projections persist). The deploy compose pg profile provides the database. Incremental resume-from-Head split out as D21b. | fixed`
- `D17 | agent-manager | agent runs can now run asynchronously, event-sourced. POST /v1/agents/{name}/run with {"async":true} records an AgentRunStarted (status "running") and queues the work, returning 202 + run_id immediately; a worker pool drains the queue, invokes the provider (with tool-calling), and records the terminal AgentRunRecorded; callers poll GET /v1/agent-runs/{run_id}. The projection folds RunStarted→running and RunRecorded→completed/failed. Crash/shutdown safety: workers process on a background context so a graceful shutdown lets in-flight runs finish (DrainWorkers bounds the wait before the log closes), and a run left "running" (started, no terminal) is re-enqueued at boot by RecoverRunning (folds the log, runs after the projections rebuild so the worker can resolve the agent). The synchronous path is unchanged. Verified: command tests (async completion, unknown-agent rejection, recovery re-enqueues + skips finished) under -race, and a real-binary async run (202→running→completed). (Streaming is still not implemented; the run is recorded as one terminal event.) | fixed`
- `D1 | eventlog | the file WAL held every envelope in memory (memory grew with the log). It now keeps only a compact in-memory byte-offset index (one int64 per record) and streams records from disk on demand via ReadAt (a single read for the requested contiguous tail) — retained memory is O(events) offsets instead of O(events × envelope size), and open scans for record boundaries without decoding every record. Records are validated lazily on Read; a corrupt complete record fails loudly there, and the projection runtime Reads the whole log at boot so corruption still stops startup (the torn-final-write recovery from D22 is preserved). Trade-off: the existing per-command log folds (caseExists / runAgentName) now read from disk rather than an in-memory slice — they were already O(n) by design; the SQLite-backed log (D18) is the indexed/scalable backend for both memory and hot reads. | fixed`
- `D12 | case-manager | SLA breaches are now pushed, not only derived on read. A sweep (command.SweepSLA, exposed as POST /v1/cases/sla-sweep for a scheduler to call) folds the case stream, finds open cases past their deadline as of now, and emits a CaseSLABreached event for each not-yet-breached one. The breach is an effect computed against the wall clock and then recorded, so replay reads the recorded breaches and stays stable; the sweep is idempotent (a breached case is skipped, and the projector folds a re-emitted breach as a no-op). The cases projection marks sla_breached + audits it. Read-time days_left / sla_state remain (the stored projection is still clock-free). | fixed`
- `D10 | web | the builder now has structured config panels for ALL node types: the nested-table types (rule = when→then repeaters, scorecard = factor when/weight rows, decision_table = mode + row when→outputs repeaters, 2d_matrix = row/col condition lists + a cell grid of literal values) joined the flat-config panels (split/connect/ai/manual_review/output/code/assignment). The raw-JSON textarea stays as the per-type advanced view. The canvas also supports drag-to-connect (Svelte Flow onconnect → addUniqueEdge, deduplicated) alongside the from/to form. The repeaters update via index-matched .map / .at (no dynamic key writes, clean under eslint security/detect-object-injection); pure helpers are unit-tested and structured editing is covered by e2e. | fixed`
- `D18 | eventlog | the file WAL is single-process, so the split-services profile gave each module its own log. Added eventlog.SQLiteLog — a shared, durable, append-only log backed by one SQLite file (global AUTOINCREMENT seq, WAL mode + busy_timeout serialize cross-process writers) behind the existing Log interface, selectable with serve --log=sqlite. SQLite has no pub/sub, so a background poller delivers newly-committed rows (its own and other processes') to in-process subscribers; Read/Head are immediately consistent and live delivery is eventual (bounded by the poll interval) — a projection-lag, not data loss. The split compose profile now shares one volume with --log=sqlite --store=sqlite so cross-component flows work across processes. Verified: two handles on one dir share events + live-deliver across handles (unit), and the real binary rebuilds projections from the shared log after restart. The monolith file WAL stays the default. | fixed`
- `D16 | agent-manager | an agent's declared tools are now executed via a tool-calling loop. The ai.Provider boundary gained tool specs, tool calls, and a conversation history; the agents loop (InvokeWithTools, bounded by maxToolSteps) lets the model request tools, executes each via a Toolbox, feeds the results back, and records the full tool trace on the run (events.ToolCall + RunView.ToolCalls) so it is auditable and replay-stable. The reference Toolbox (agent-manager/tools.ConnectorToolbox) exposes Context Layer connectors as tools and is wired into both the Agent Manager and the engine AI node at the composition root. The OpenAI-compatible HTTP provider maps tools↔tool_calls (verified against a mock server); the Stub still answers directly (no tools). | fixed`
- `D14 | context-layer | added a SQL reference connector (connector type "sql") alongside http + mock_bureau. Config is {dsn, query, args[]}; it opens the configured database (the pure-Go modernc.org/sqlite driver, already a dependency — "sqlite" is the only built-in driver, others fail loudly), runs the query binding the declared args as named parameters (values, never string-interpolated — caller params cannot inject SQL), and returns {"rows":[...]} bounded to 1000 rows under the fetch timeout. Verified end to end against a temp sqlite db. Other drivers (Postgres/MySQL) become available in a build that imports them, behind the same Connector interface. | fixed`
- `D19 | decision-engine | the JSON-Schema validator (platform/schema, shared by decide-input contracts and agent structured output) was a flat object/required/per-property-type check; richer keywords were accepted but unenforced. It is now a recursive validator covering: type (incl. type arrays), required, properties, additionalProperties (bool or schema), min/maxProperties, enum, const, allOf/anyOf/oneOf/not, numeric minimum/maximum/exclusive bounds/multipleOf, string minLength/maxLength/pattern and formats (email/uuid/date/date-time), array items/minItems/maxItems/uniqueItems, and local $ref (a JSON pointer into the same document, e.g. #/$defs/X). Recursion is depth-bounded (fails loudly on a $ref cycle). Unknown keywords stay lenient. | fixed`
- `D15 | context-layer | the http connector dialed an operator-configured URL with no SSRF protection. It now enforces an egress policy at dial time (a net.Dialer Control hook that inspects the resolved IP, so it guards every redirect hop and resists DNS rebinding): loopback, RFC1918/ULA private, link-local, unspecified, and multicast targets are refused by default. Operators whose connectors legitimately reach internal hosts opt in with INTRAKTIBLE_CONNECTOR_ALLOW_PRIVATE (logged loudly at boot). connectors.InvokeWith / Provider.Egress / service.WithEgress thread the policy; the scheme + time + size bounds remain. | fixed`
- `D20 | web/auth | the UI has a login page (web/src/routes/login): an API key is exchanged for a session cookie, the home page shows the signed-in identity (via GET /v1/me) with sign-out, and sessions are durable (see the D7/storesessions work) so they survive a restart under --store=sqlite. The module pages still offer an API-key input as well — both auth methods work; a full migration of those pages to cookie-only is optional future polish. | fixed`
- `D13 | web | the case-detail page now renders the case context (the source decision's inputs / agent escalation reference) as a readable key-value view (lib/kv.displayEntries: primitives as text, nested values as compact JSON) instead of nothing. A fully schema-aware decision-inputs view remains a possible future polish. | fixed`
- `D17 (partial) | agent-manager | a schema-constrained agent's structured output is now validated against the agent's schema (shared platform/schema.ValidateObject, the same JSON-Schema subset the decide-input check uses); a mismatch is a recorded failed run (fail loudly). Remaining (async/queued/streaming runs) stays open as D17. | fixed`
- `D5 | ai | added a real OpenAI-compatible HTTP provider (ai.NewHTTP — POST {base}/chat/completions, Bearer auth, json_object response_format when a schema is requested) behind the existing ai.Provider interface. It works with OpenAI/Ollama/vLLM/gateways and Anthropic-compatible endpoints. Wired in via INTRAKTIBLE_AI_BASE_URL/_API_KEY/_MODEL/_PROVIDER (Stub stays the fallback). Verified text + structured + error paths against a mock server and end-to-end on the real binary. | fixed`
- `D22 | eventlog | crash safety: WAL.load now recovers a torn final record (a crash mid-Append leaves a trailing line with no newline) by truncating it on reopen, instead of failing to open and bricking the log. No acknowledged event is dropped (Append fsyncs before returning), and a corrupt complete record mid-file still fails loudly. (D1's in-memory/segmentation scaling is still open.) | fixed`
- `D3 | projection | a stalled projection (a live-apply error stopped the consumer) is now surfaced: GET /healthz returns 503 "degraded" with the error when the runtime's Err() is set, so an orchestrator can restart/depool the node (a fresh boot rebuilds the projections). Dead-lettering/auto-skip is intentionally NOT added — silently dropping an event would violate fail-loudly and leave projections inconsistent. | fixed`
- `D2 | store | added a durable SQLite projection store (store.NewSQLite, pure-Go modernc.org/sqlite — no CGO) behind the existing store.Store interface, selectable with serve --store=sqlite (persists to <data-dir>/projections.db, WAL + busy_timeout for one writer / many readers). Verified data survives a restart. Postgres adapter split out as D21. | fixed`
- `D7 | auth | added a login flow: POST /v1/login exchanges a valid API key for an HttpOnly session cookie (the Authenticate middleware already accepted it), POST /v1/logout revokes it, GET /v1/me returns the caller; sessions now expire (DefaultSessionTTL) and can be revoked.
