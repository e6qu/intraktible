## Fixed
- `D21b | store/projection | durable stores now RESUME from a checkpoint instead of fully rebuilding from offset 0 on every boot. The store gained an atomic transaction (store.Tx / store.TxStore, implemented by SQLite + Postgres; a Tx satisfies store.Store so projectors write through it unchanged); the projection runtime applies each event AND advances a checkpoint (_projection/applied_head) in one transaction, so the non-idempotent projectors (analytics counters, case audit/notes appends, run/event counters) are applied exactly once — no per-projector idempotency required. At boot it resumes from checkpoint+1 (transactional durable store with a valid checkpoint) or full-rebuilds (memory, or first boot); an in-memory applied-watermark also de-dups the boot-rebuild/live-bus overlap. Verified: a durable-store resume test proves no re-apply and no reset (a sentinel survives), and a real-binary --store=postgres restart resumed with the checkpoint row persisted. | fixed`
- `D21 | store | added a Postgres store.Store adapter (store.NewPostgres via jackc/pgx/v5 + pgxpool) behind the existing interface, selectable with serve --store=postgres (DSN from INTRAKTIBLE_POSTGRES_DSN) — the durable/shared option for large or multi-process projections. - `D17 | agent-manager | agent runs can now run asynchronously, event-sourced. The projection folds RunStarted→running and RunRecorded→completed/failed. Crash/shutdown safety: workers process on a background context so a graceful shutdown lets in-flight runs finish (DrainWorkers bounds the wait before the log closes), and a run left "running" (started, no terminal) is re-enqueued at boot by RecoverRunning (folds the log, runs after the projections rebuild so the worker can resolve the agent). - `D1 | eventlog | the file WAL held every envelope in memory (memory grew with the log). It now keeps only a compact in-memory byte-offset index (one int64 per record) and streams records from disk on demand via ReadAt (a single read for the requested contiguous tail) — retained memory is O(events) offsets instead of O(events × envelope size), and open scans for record boundaries without decoding every record. Records are validated lazily on Read; a corrupt complete record fails loudly there, and the projection runtime Reads the whole log at boot so corruption still stops startup (the torn-final-write recovery from D22 is preserved).
