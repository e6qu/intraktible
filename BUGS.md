## Fixed
- `D21b | store/projection | durable stores now RESUME from a checkpoint instead of fully rebuilding from offset 0 on every boot. The store gained an atomic transaction (store.Tx / store.TxStore, implemented by SQLite + Postgres; a Tx satisfies store.Store so projectors write through it unchanged); the projection runtime applies each event AND advances a checkpoint (_projection/applied_head) in one transaction, so the non-idempotent projectors (analytics counters, case audit/notes appends, run/event counters) are applied exactly once — no per-projector idempotency required. At boot it resumes from checkpoint+1 (transactional durable store with a valid checkpoint) or full-rebuilds (memory, or first boot); an in-memory applied-watermark also de-dups the boot-rebuild/live-bus overlap. Verified: a durable-store resume test proves no re-apply and no reset (a sentinel survives), and a real-binary --store=postgres restart resumed with the checkpoint row persisted. | fixed`
- `D21 | store | added a Postgres store.Store adapter (store.NewPostgres via jackc/pgx/v5 + pgxpool) behind the existing interface, selectable with serve --store=postgres (DSN from INTRAKTIBLE_POSTGRES_DSN) — the durable/shared option for large or multi-process projections. One docs(collection, key, doc JSONB) table, parameterized queries. VERIFIED against a real PostgreSQL 17 (not just written): the full Store-contract test (store_test.go TestPostgresStore, gated on INTRAKTIBLE_TEST_POSTGRES so the no-Postgres CI skips it) passes, and a real-binary --store=postgres smoke created a flow + read it back after a restart (projections persist). - `D17 | agent-manager | agent runs can now run asynchronously, event-sourced. POST /v1/agents/{name}/run with {"async":true} records an AgentRunStarted (status "running") and queues the work, returning 202 + run_id immediately; a worker pool drains the queue, invokes the provider (with tool-calling), and records the terminal AgentRunRecorded; callers poll GET /v1/agent-runs/{run_id}. The projection folds RunStarted→running and RunRecorded→completed/failed. Crash/shutdown safety: workers process on a background context so a graceful shutdown lets in-flight runs finish (DrainWorkers bounds the wait before the log closes), and a run left "running" (started, no terminal) is re-enqueued at boot by RecoverRunning (folds the log, runs after the projections rebuild so the worker can resolve the agent). The synchronous path is unchanged. Verified: command tests (async completion, unknown-agent rejection, recovery re-enqueues + skips finished) under -race, and a real-binary async run (202→running→completed). (Streaming is still not implemented; the run is recorded as one terminal event.) | fixed`
- `D1 | eventlog | the file WAL held every envelope in memory (memory grew with the log). It now keeps only a compact in-memory byte-offset index (one int64 per record) and streams records from disk on demand via ReadAt (a single read for the requested contiguous tail) — retained memory is O(events) offsets instead of O(events × envelope size), and open scans for record boundaries without decoding every record. Records are validated lazily on Read; a corrupt complete record fails loudly there, and the projection runtime Reads the whole log at boot so corruption still stops startup (the torn-final-write recovery from D22 is preserved). Trade-off: the existing per-command log folds (caseExists / runAgentName) now read from disk rather than an in-memory slice — they were already O(n) by design; the SQLite-backed log (D18) is the indexed/scalable backend for both memory and hot reads. | fixed`
- `D12 | case-manager | SLA breaches are now pushed, not only derived on read. The breach is an effect computed against the wall clock and then recorded, so replay reads the recorded breaches and stays stable; the sweep is idempotent (a breached case is skipped, and the projector folds a re-emitted breach as a no-op).
